# 2026-06-16 main.py 动态蒙版改为逐路串行学习

## 修改日期

2026-06-16

## 修改目标

根据板端验证结果，动态蒙版即使做了 10 秒错峰，启动后仍会在后续阶段卡住。  
本次改动不再让多路相机的动态蒙版学习阶段互相重叠，而是改成“前一路学完，后一路再开始”。

## 本次改动内容

### 1. 动态蒙版从错峰并行改为串行学习

现在每一路相机的动态蒙版学习开始时间不再按固定 10 秒间隔启动，而是按“前一路完整学习窗口结束后，再启动下一路”的方式调度。

当前参数：

```python
DYNAMIC_MOTION_MASK_SECONDS = 60.0
DYNAMIC_MOTION_MASK_START_STAGGER_SECONDS = 0.0
DYNAMIC_MOTION_MASK_SERIAL_GAP_SECONDS = 0.0
```

因此 5 路相机的启动时序变为：

- Cam0：0 秒开始，60 秒结束
- Cam1：60 秒开始，120 秒结束
- Cam2：120 秒开始，180 秒结束
- Cam3：180 秒开始，240 秒结束
- Cam4：240 秒开始，300 秒结束

这意味着任意时刻都只有一路相机处于动态蒙版学习状态。

### 2. 保留多进程默认路径

本次不恢复多进程开关，仍保持：

- 优先使用多进程摄像头 worker
- 若环境不支持 `fork`，自动回退线程模式

## 影响范围

本次修改影响：

- 动态蒙版学习时序
- 板端启动后数分钟内的动态蒙版可用时间分布
- 启动阶段的 CPU / IPC / 编码叠加压力

本次未修改：

- 帧差生成逻辑
- 静态背景蒙版逻辑
- ROI 候选生成逻辑
- YOLO 推理方式
- 轨迹确认与绿框逻辑

## 预期代价

串行学习可以最大限度避免多路动态蒙版学习重叠，但代价是后面的相机需要更久才会拿到最终动态蒙版。

例如 Cam4 要到约 300 秒后才完成动态蒙版学习。

## 后续验证重点

板端下一步重点观察：

1. 串行学习后是否还会出现“启动几秒正常、随后卡死”的问题
2. Cam0 到 Cam4 的 ready 日志是否按 60 秒步进出现
3. 在后几路动态蒙版尚未 ready 前，检测是否还能维持基本可用
