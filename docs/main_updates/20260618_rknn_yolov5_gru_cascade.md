# 2026-06-18 main_yolov5_gru.py 适配 RKNN 与 YOLOv5 + GRU 级联跟踪

## 修改日期

2026-06-18

## 修改目标

在板端主代码 main.py 的基础上创建副本 main_yolov5_gru.py，并在其中整合 development 分支的 YOLOv5 + GRU 级联跟踪与轨迹预测方案，同时将输入分辨率适配为 1080p (1920x1080)。这样可在保留原有 main.py 完全不被修改的前提下，让高层板端运行全新的 YOLOv5 + GRU 算法，且保留原有的多摄调用、通信协议与数传等逻辑。

## 本次改动内容

### 1. 分辨率适配为 1080p
- 修改 `CAPTURE_W, CAPTURE_H = 1920, 1080`。
- 推理和取流全部基于 1080p 分辨率。

### 2. 新增关键算法模块类
- **`UAVTrajectoryNet` (GRU 预测网络)**：用于轨迹分类（区分 UAV 与噪点）和未来 5 步轨迹偏移量预测。在板端以 CPU 运行推理。
- **`FeatureTracker` (特征跟踪器)**：管理目标轨迹历史缓冲区，提取特征，判定跟踪状态和死锁/丢失状态（例如静态死锁判定、连续 12 帧未更新触发重置等）。
- **`MockFallbackImpl` (CPU 帧差候选生成器)**：当全局检测未检测到目标时，利用轻量级 CPU 差分检测生成最多 3 个 256x256 的候选 ROI 区域，送入 NPU 进一步分类。

### 3. 多模态 YOLO 级联检测 (Camera Job & NPU Worker)
- **ROI 搜索模式**：若当前有跟踪目标，在其 1080p 映射坐标周围裁剪 $640\times640$ 区域送至 NPU 推理。
- **Fallback 候选模式**：若当前目标丢失，使用 `MockFallbackImpl` 帧差筛选出最多 3 个高疑似运动候选点，分别裁剪 $256\times256$ 并双线性插值缩放到 $640\times640$ 送至 NPU 确认。
- **Global 全局检索模式**：若没有候选 ROI，则将整张 1080p 图像缩放到 $640\times640$ 注入 NPU 检索。

### 4. 跨分辨率坐标无损对齐 (2K Tracker <=> 1080p Real-time)
- GRU 网络和 Feature Tracker 在训练时基于 2K 空间（$2560\times1440$）。为了不重训模型，采用**坐标重映射**：
  - NPU 检测出的 1080p 坐标映射为 2K 坐标后送入 `FeatureTracker`。
  - `FeatureTracker` 进行卡尔曼式更新，并提取 8 维时序特征送入 GRU。
  - GRU 预测出的未来 5 步 2K 偏移量，再被重映射回 1080p 坐标，用于画面绘制和云台控制通信。

### 5. 轨迹判定与可视化增强
- **噪点过滤**：GRU 对轨迹进行二分类，输出概率 `< 0.5` 视为噪点（画红框且不向云台发数），`>= 0.5` 确认是真正的无人机（画绿框且向云台发送数传包）。
- **轨迹渲染**：在推流预览画面中，绘制绿色的历史轨迹线，以及黄色的未来 5 步预测轨迹点和折线。

## 影响范围与规则确认

- **帧差逻辑**：副本 `main_yolov5_gru.py` 代替了原 `main.py` 中的精细背景差分、背景模型和 LCM 滤波，改用新方案的无状态 `MockFallbackImpl` 亮点定位进行局部 ROI 候选点快速定位，不再使用复杂的背景建模，极大节省了板端 CPU 开销。原 `main.py` 文件完全保留未被改动。
- **YOLO 推理**：继续调用板端 `yololib.py` 初始化 `YoloRKNN`，基于 1080p 级联 ROI 裁剪输入。
- **数据发送协议**：保留原有 UDP Telemetry 数传格式 `[x1, y1, x2, y2, range]` 和云台指令，不受新跟踪器影响。

## 默认参数变化

- 摄像头分辨率：由 2K (`2560x1440`) 降为 1080p (`1920x1080`)。
- 跟踪机制：由原基于帧差及 `TrajectoryFilter` 改为级联 YOLOv5 + GRU (`FeatureTracker`) 机制。
- 版本标识修改：
  - `ALGORITHM_VERSION = "rknn-yolov5-gru-cascade-20260618"`
