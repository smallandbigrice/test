# 2026-06-18 main_gru_detect.py 纯 CPU 灰度二值化检测与 GRU 级联跟踪

## 修改日期

2026-06-18

## 修改目标

创建副本文件 `main_gru_detect.py`，并在其中整合 `development` 分支中 `gru detect` 模块的纯 CPU 无人机目标检测、时序追踪以及 GRU 轨迹噪点过滤/预测方案。该方案全程不使用 YOLOv5 NPU 推理，从而不需要加载任何 RKNN 模型，极大降低系统对 NPU 的依赖，并利用 CPU 实现高帧率、低能耗的多摄检测。同时，原始 `main.py` 保持完全未改动。

## 本次改动内容

### 1. 分辨率与输入类型适配
- 修改 `CAPTURE_W, CAPTURE_H = 1920, 1080`（1080p 分辨率）。
- 摄像头取流直接基于 1080p 进行处理。同时全面适配灰度视频输入：当摄像头本身输出的就是单通道灰度视频时，程序会自动捕获并避免多余的灰度转换，极大节省了板端 CPU 开销。

### 2. 纯 CPU 灰度二值化目标检测层
- 放弃原 NPU YOLOv5 推理。
- 引入 `FeatureTracker` 中的 `detect_all_centroids` 方法：
  - 对输入 1080p 图像提取灰度图（若已经是单通道灰度图，则直接跳过转换），应用绝对灰度反向二值化（`cv2.threshold` 反向阈值法，默认 `thresh_val=120`）。
  - 通过 `cv2.findContours` 提取轮廓并过滤面积在 `1` 到 `2000` 像素之间的连通域作为候选点中心。
  - 引入邻近连通域融合去重机制，将物理间距小于 `20px` 的微小区域进行加权质心融合，融合成单个候选框，作为多目标追踪器的更新输入。

### 3. 多目标特征追踪器与防抖 (CPU)
- 引入自适应 1080p 分辨率的 `FeatureTracker` 和 `Track` 轨迹段管理类：
  - **近邻匹配与外推**：使用单帧最大运动位移门限（`max_drone_move = 60px` 乘以 1080p 缩放因子）将检测质心与已锁定的轨迹进行最近邻贪婪匹配。未匹配到的轨迹进行惯性外推以保持轨迹链。
  - **冲突去重合并**：如果两条轨迹的物理显示间距小于 `35px`，则自动判定为重复检测并进行合并。
  - **背景死锁阻断**：当轨迹连续 30/50 帧在地面区域（`sky_height_limit` 以下）微小范围抖动时，自动判定为建筑物静态粘滞死锁并予以销毁。
  - **防抖滤波**：引入一阶指数低通滤波器（`alpha = 0.65`）对无人机的物理显示框进行防抖修平。

### 4. GRU 时序噪点过滤与未来轨迹预测
- 时序滑动窗口长度设为 20 帧，输入特征维度为 8 维。
- 当某条轨迹历史长度达到 20 帧时，在 CPU 上运行 `UAVTrajectoryNet`（GRU 模型）：
  - **真伪 UAV 判决**：通过轨迹分类概率过滤森林边缘、树木晃动等弱噪声。概率 $\geq0.60$ 确认为 true UAV，概率 $<0.60$ 过滤为噪点。
  - **未来轨迹预测**：预测无人机未来 5 帧的飞行偏移，叠加当前防抖平滑坐标后在画面上绘制黄色预测轨迹。

### 5. 多摄取流与云台 UDP 发数
- 多路相机的 `capture_job` 直接在 CPU 上独立并行运行检测追踪。
- 保留 `main.py` 的 UDP 云台数据发送协议和预览流推流发送。只有在当前路相机判定出唯一的黄金主轨迹（`primary_track`）且为真实 UAV 时，才发送电传数据 `[x1, y1, x2, y2, range]`，有效防止误检测引起云台抖动。

## 影响范围

- **NPU 使用**：移除了 `inference_worker`，完全不调用 NPU（不使用 RKNN）。
- **帧差逻辑**：用更高效的绝对灰度值二值化连通域检测代替了原 `main.py` 复杂的历史建模背景帧差和 LCM 过滤，极大地释放了 CPU 运行压力。
- **原始 `main.py`**：保持完全原样，未做任何修改。
